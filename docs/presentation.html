Шаг 1 — архитектура парсеров

BaseParser: один интерфейс для всех

Все 11 парсеров наследуют от BaseParser. Каждый обязан реализовать один метод — всё остальное (хэш, статус, время) добавляется автоматически.

код 
class BaseParser(ABC):

    source_name: str = "unknown"
    channel: str = "ru"

    @abstractmethod
    def fetch(self) -> list[dict]:
        ...

    def run(self) -> list[dict]:
        raw = self.fetch()
        prepared = [
            self.prepare(v)
            for v in raw
        ]
        return prepared
по-русски
Базовый класс. ABC = «абстрактный» — нельзя создать напрямую, только через наследника.
Каждый парсер объявляет своё имя ("remotive") и канал публикации — "ru" или "global".
@abstractmethod — контракт: любой наследник ОБЯЗАН написать fetch(). Иначе — ошибка при запуске.
fetch() — единственное, что пишет автор каждого нового парсера.
run() вызывает fetch(), добавляет хэш / статус / время к каждой вакансии. Пишется один раз — работает для всех 11.
Шаг 1 — реальный парсер

Реальный пример: парсер Remotive

Remotive — самый простой парсер: публичный API без ключей, возвращает JSON. Посмотрим как он устроен.

код 
class RemotiveParser(BaseParser):
    source_name = "remotive"
    channel = "global"

    def fetch(self) -> list[dict]:
        resp = requests.get(API_URL, timeout=15)
        resp.raise_for_status()
        jobs = resp.json().get("jobs", [])
        result = []
        for job in jobs:
            title = job.get("title", "")
            if not _is_relevant(title):
                continue
            result.append({
                "external_id": str(job["id"]),
                "title": title,
                "url": job.get("url", ""),
                ...и ещё 6 полей
            })
        return result
по-русски
Говорим: «я парсер Remotive, публикую в глобальный канал».
Делаем HTTP-запрос к API Remotive. timeout=15 — ждём не дольше 15 секунд.
raise_for_status() — если сервер вернул ошибку (404, 500), бросаем исключение сразу. BaseParser.run() поймает его и вернёт пустой список.
Берём массив "jobs" из JSON. Если ключа нет — пустой список, не падаем.
Для каждой вакансии: если заголовок не содержит researcher/ux/cx — пропускаем сразу, не тратим память.
Собираем нужные поля в dict. BaseParser.prepare() добавит хэш, source, channel, parsed_at.

Шаг 5 — надёжность AI

Цепочка fallback: три модели про запас

Если первая модель не отвечает или возвращает мусор — система переключается на следующую. Ни одна вакансия не остаётся без скора.

Цепочка приоритетов
1 Gemini 2.5 Flash Lite
google/gemini-2.5-flash-lite · быстрый, дешёвый
2 Mistral Small 3.1
mistralai/mistral-small-3.1 · резервный
3 Llama 3.3 70B
meta-llama/llama-3.3-70b:free · бесплатный
Как это работает в коде
for model in sorted(MODELS, key=...priority):
    try:
        response = client.chat.completions.create(
            model=model["id"],
            response_format={"type": "json_object"},
            temperature=0,
            timeout=15.0,
        )
        return json.loads(response.choices[0].message.content)
    except Exception as exc:
        last_exc = exc  # пробуем следующую

raise RuntimeError("All LLM models failed")
⚙️ Параметры запроса
json_object — AI обязан вернуть валидный JSON, не текст
temperature=0 — детерминированный ответ, без фантазий
timeout=15s — если модель тупит — не ждём, идём к следующей

Шаг 6 — Telegram API

Inline-кнопки: как это работает

Кнопки под сообщением — это inline keyboard. При нажатии Telegram отправляет боту callback_data — строку с командой и ID вакансии.

Создание кнопок (moderator.py)
def _keyboard(vacancy_id: int, channel: str):
    return {
        "inline_keyboard": [[
            {
                "text": "✅ Опубликовать",
                "callback_data":
                    f"approve:{vacancy_id}:{channel}"
            },
            {
                "text": "❌ Отклонить",
                "callback_data":
                    f"reject:{vacancy_id}"
            },
        ]]
    }
Разбор нажатия (handlers.py)
parts = query.data.split(":")
action    = parts[0]       # "approve" / "reject"
vacancy_id = int(parts[1])  # ID вакансии из БД
actor = query.from_user.first_name
Жизненный цикл нажатия
1. Модератор нажимает ✅
Telegram отправляет боту callback_query с data = "approve:42:global"
2. Бот отвечает answer()
Обязательно в течение 60 сек. — иначе Telegram покажет «таймаут». Убирает спиннер на кнопке.
3. Парсит data → действие
"approve:42:global" → action=approve, id=42, channel=global
4. Публикует пост
send_message в TELEGRAM_CHANNEL_GLOBAL → mark_posted(42) → убирает кнопки
⚠️ callback_query живёт 60 сек.

После передеплоя старые кнопки выглядят рабочими, но query уже мёртв. Бот это обрабатывает: пишет модератору «дождитесь следующего цикла».

Схема базы данных

Все вакансии хранятся в таблице vacancies. Уникальность — по hash. Ни одна вакансия не теряется: отклонённые тоже попадают в БД со статусом rejected.

Таблица vacancies
CREATE TABLE vacancies (
  id          SERIAL PRIMARY KEY,
  external_id TEXT,
  source      TEXT,       -- "hh", "remotive"...
  title       TEXT,
  company     TEXT,
  salary_min  INTEGER,
  salary_max  INTEGER,
  currency    TEXT,
  location    TEXT,
  work_format TEXT,
  url         TEXT,
  description TEXT,
  snippet     TEXT,
  hash        TEXT UNIQUE,  -- SHA-256
  status      TEXT DEFAULT 'new',
  channel     TEXT,       -- "ru" / "global"
  parsed_at   TIMESTAMP,
  posted_at   TIMESTAMP
);
Ключевые поля
hash TEXT UNIQUE
SHA-256 от source|external_id или title|company|url. Дубли → INSERT IGNORE
status TEXT
new → pending → posted / rejected
channel TEXT
Определяет, в какой Telegram-канал отправить после одобрения
description TEXT
Полный HTML текст вакансии. Для Adzuna — обогащается через scraping страницы
PostgreSQL, не SQLite

На Railway — полноценный PostgreSQL. Подключение через DATABASE_URL env-переменную, драйвер psycopg2.

Инфраструктура — планировщик

APScheduler: сердце системы

Всё начинается с BlockingScheduler — он держит процесс живым и запускает run_cycle() каждые N минут.

scheduler.py — точка входа
def start_scheduler():
    database.init_db()

    # первый запуск сразу
    run_cycle()

    scheduler = BlockingScheduler(
        timezone="Europe/Moscow"
    )
    scheduler.add_job(
        run_cycle,
        "interval",
        minutes=config.PARSE_INTERVAL_MINUTES,
        id="main_cycle",
        max_instances=1,  # только один
    )
    scheduler.start()  # блокирует поток
Детали
BlockingScheduler
Держит основной поток занятым — идеально для Railway, где контейнер должен «что-то делать»
max_instances=1
Если прошлый цикл ещё не завершился — новый не стартует. Защита от накопления очередей
Первый запуск немедленно
run_cycle() вызывается до старта планировщика — не нужно ждать N минут после деплоя
PARSE_INTERVAL_MINUTES
Задаётся через env-переменную в Railway. Сейчас — 30 минут

Миграции: схема без потери данных

SQL-миграции в папке migrations/ применяются вручную один раз. Позволяют расширять схему не удаляя данные.

История миграций
001_vacancy_scores.sql
Создание таблицы vacancy_scores с 3 индексами. Отделила результаты AI от самих вакансий.
002_add_model_tracking.sql
Добавлены поля model_used и latency_ms — для мониторинга какая модель отвечала.
003_add_post_enrichment.sql
Поле post_enrichment JSONB — хранение AI-дополнений к посту (summary, ключевые задачи, требования).
Пример миграции
-- 003_add_post_enrichment.sql
ALTER TABLE vacancy_scores
  ADD COLUMN IF NOT EXISTS
    post_enrichment JSONB;
IF NOT EXISTS — ключевой паттерн

Каждая миграция идемпотентна — можно запустить дважды без ошибки. Безопасно при переезде на новый сервер.

Применение вручную

psql $DATABASE_URL -f migrations/003.sql — один раз после деплоя. Автоматической накатки пока нет.
